# 需求成熟度模型

## 需求成熟度阶梯

| 层级 | 名称 | 特征 | 典型表达 |
|------|------|------|---------|
| L0 | 冲动 | 纯 FOMO，没有具体想法 | 「我也要搞 AI！」「大家都在用 agent」 |
| L1 | 欲望 | 知道想要什么结果，不知道路径 | 「想赚钱」「想要数字分身」「想提效」 |
| L2 | 方向 | 知道大致方向，缺少具体场景 | 「想自动化内容生产」「提升客服效率」 |
| L3 | 场景 | 有角色+动作，缺约束和边界 | 「每天给客户出 20 张海报」 |
| L4 | 需求 | 有明确输入/输出/约束/验收标准 | 「接受品牌规范，输出海报，日产 50 张」 |

**目标：将用户从当前层级引导到 L4。**

---

## 诊断信号表

### 层级诊断

| 信号 | 判定层级 | 示例 |
|------|---------|------|
| 「我也要」「别人都在」「很火」 | L0 冲动 | 「AI agent 很火，我也想搞一个」 |
| 「想要」「希望」+ 抽象结果 | L1 欲望 | 「想让 AI 帮我赚钱」 |
| 有动词但缺主语/宾语/场景 | L2 方向 | 「想自动化内容」「提升效率」 |
| 有角色+动作，缺约束 | L3 场景 | 「让 AI 帮我回复客户消息」 |
| 有输入/输出/约束/验收标准 | L4 需求 | 「系统接受商品问题，3秒内回复标准答案」 |

### 方案污染检测

以下信号表明用户在定义问题前已嵌入解决方案：

| 信号 | 示例 |
|------|------|
| 提到具体工具名 | 「用龙虾/Dify/n8n 来做…」 |
| 提到技术方案 | 「用多 agent/RAG/微调来…」 |
| 「用 X 来做 Y」句式 | 「用 AI agent 做客服」 |
| 先说方案再说问题 | 「我想搭一个工作流，因为…」 |

**处理原则：** 不否定用户的方案想法，先记录，然后温和引导回到问题本身。

## 逐层下降策略

### L0 → L1（从冲动到欲望）

**核心问题：** 找到触发冲动的真实吸引力。

提问模板：
- 「你看到别人用 AI 做了什么，让你觉得特别心动？」
- 「如果 AI 能帮你解决一个工作/生活中的烦恼，你第一个想到什么？」

选择题备用（用户答不出时）：
```
你最想用 AI 解决哪类问题？
A. 重复性工作太多，想省时间
B. 想拓展收入来源
C. 想提升内容/创作质量
D. 其他（请描述）
```

### L1 → L2（从欲望到方向）

**核心问题：** 从结果倒推路径。

提问模板：
- 「假设半年后这个目标实现了，回头看，AI 具体帮你做了什么？描述一个典型的一天。」
- 「你目前每天花最多时间在什么工作上？」

选择题备用：
```
基于你提到的 [目标]，以下哪个方向最接近你想做的？
A. [方向1 — 根据上下文生成]
B. [方向2]
C. [方向3]
D. 都不是，我想做的是…
```

### L2 → L3（从方向到场景）

**核心问题：** 锚定现状和痛点。

提问模板：
- 「现在这件事是谁在做？具体怎么做的？」
- 「整个流程里，哪个环节最让你头疼？」
- 「上一次因为这个问题浪费时间/金钱是什么时候？发生了什么？」

### L3 → L4（从场景到需求）

**核心问题：** 明确边界和验收标准。

提问模板：
- 「这个场景里，输入是什么？最终的产出是什么？」
- 「做成什么样你会满意？什么样算不合格？」
- 「有没有时间、预算、或者技术上的限制？」
- 「一天/一周需要处理多少量？」

## 下降受阻：资源盘点策略

当用户无法回答开放性问题（典型回答：「我不知道」「你告诉我」「都行」）时，切换策略：

**不问「你想做什么」，改问「你有什么」：**

```
我换个角度——先聊聊你的情况：
· 你有什么技能或专长？（设计/写作/销售/编程/运营/其他）
· 你有什么现成资源？（粉丝群/客户/内容库/产品/人脉）
· 你每天能投入多少时间在这件事上？
· 你能接受的启动成本大概是多少？
```

然后基于用户的资源画像，**主动建议 2-3 个匹配的方向**，让用户选择。选完后回到正常下降流程。

## 多需求拆包

识别信号：一句话里出现多个不相关的动词/场景，用「和」「以及」「还有」连接。

处理步骤：
1. 列出识别到的每个独立需求
2. 问用户：「这几件事之间有关联吗？还是各自独立的？」
3. 如果独立：用 AskUserQuestion 让用户选最先要做的
4. 如果有关联：找出核心需求和衍生需求，从核心开始

## 三重校验细节

### 可行性校验

完成六问后，评估需求的技术可行性：
- 目前的 AI/技术能力能否实现？
- 如果不能完全实现，哪些部分可以？哪些需要人工兜底？
- 坦诚告知限制，不夸大 AI 能力

模板：
> 「关于你的需求，有一点我想坦诚说明：[X 部分] 目前技术上还做不到 / 效果有限。
> 但 [Y 部分] 完全可以实现。我建议先把 Y 做好，X 后续随技术进步再迭代。」

### 矛盾检测

常见矛盾模式：
- 速度 vs 质量（「又快又好」）
- 成本 vs 效果（「免费但要专业级」）
- 简单 vs 功能（「简单易用但功能全面」）
- 自动 vs 可控（「全自动但我要能控制每个细节」）

发现矛盾时：
> 「我注意到两个需求之间有些矛盾：[A] 和 [B] 很难同时做到。
> 如果必须选一个优先，你更看重哪个？」

### 需求重构

当用户描述的是「解决方案」而非「根本问题」时，温和提出重构：

> 「你说想要 [用户的方案]，我理解你的出发点。
> 不过我在想，你真正想解决的问题是不是 [重构后的问题]？
> 如果是的话，可能有更直接的方式。你觉得呢？」

**注意：** 重构是建议，不是否定。如果用户坚持原始需求，尊重用户的选择。
