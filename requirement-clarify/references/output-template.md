# 需求文档输出模板

## 完整文档结构

```markdown
# 需求澄清文档

## 基本信息

- **需求提出者**：[用户名/角色]
- **原始描述**：「[用户原话]」
- **诊断结果**：[层级] + [是否方案污染]
- **澄清日期**：[日期]

---

## 一句话需求

[用一句话概括最终澄清后的核心需求]

---

## 需求画像

| 维度 | 内容 |
|------|------|
| **WHO — 谁在用** | [用户角色、使用者画像] |
| **AS-IS — 现状** | [现在怎么做的，用什么工具，什么流程] |
| **PAIN — 痛点** | [最核心的问题是什么，具体影响] |
| **WHY — 动机** | [为什么要解决，预期收益/ROI] |
| **DONE — 成功标准** | [做成什么样算成功，可验收的指标] |
| **LIMIT — 约束** | [预算/时间/技术/人力/合规等限制] |

---

## 校验结果

### 可行性
[技术上能否实现？哪些部分可行，哪些有限制？]

### 矛盾检测
[需求之间是否有冲突？如何取舍？]

### 需求重构
[是否发现了更本质的问题？用户是否认同重构？]

---

## 技术方案推介

### 方案 A：[方案名]

**一句话解释**
[用大白话，不含任何技术术语]

**生活类比**
[用日常生活中的事物类比，帮助理解原理]

**优点**
- [用用户关心的维度描述]

**缺点**
- [同上]

**适合你如果**
[什么样的情况下选这个方案最合适]

<details>
<summary>技术细节（给开发参考）</summary>

[技术架构、工具链、关键组件、实现路径]

</details>

---

### 方案 B：[方案名]

[同上结构]

---

### 方案 C：[方案名]（如有必要）

[同上结构]

---

### 方案对比

| 你最关心的 | 方案 A | 方案 B | 方案 C |
|-----------|--------|--------|--------|
| 上手难度 | | | |
| 每月成本 | | | |
| 出品质量 | | | |
| 灵活性 | | | |
| 见效速度 | | | |
| 扩展性 | | | |

> **维度选择原则：** 从六问中提取用户最关心的 4-6 个维度，不要用技术维度（如「可扩展性」「解耦程度」）。
> 用用户能感知的维度（如「上手要多久」「每月花多少钱」「效果好不好」）。

---

### 推荐方案

**推荐 [方案X]**

**理由：**
1. [基于用户具体情况的理由1]
2. [理由2]
3. [理由3]

**为什么不选其他方案：**
- 方案 Y：[现阶段不适合的原因，但说明未来什么时候可以切换]
- 方案 Z：[不适合的原因]

---

## 下一步

- [ ] 用户确认需求文档
- [ ] 用户确认技术方案选择
- [ ] 进入 brainstorming skill 做技术设计
- [ ] [其他待办，如有]
```

---

## 使用说明

- 不是所有字段都必填——根据实际对话内容填写，没聊到的留空或标注「未涉及」
- 方案数量根据实际情况决定（2-3 个为宜），不必凑数
- 对比表的维度从六问中提取，每次不同——用户关心成本就有成本，关心速度就有速度
- 文档语气保持亲切专业，像在和朋友聊天，不像在写合同
