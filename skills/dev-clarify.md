# /dev-clarify — 需求澄清

**dev-go 的前置 skill**。需求模糊时先跑这个，产出 dev-go 可直接执行的需求文档。

> 需求已经清楚？→ 跳过，直接用 `dev-go`。

## 流程

| Phase | 做什么 | 说明 |
|-------|--------|------|
| 0. 层级诊断 | 判断需求在 L0-L4 哪层，检测方案污染 | L0=冲动，L4=可执行。见 references/maturity-model.md |
| 1. 方案分离 | 检测到"用 X 做 Y"时，先剥离方案，回到问题本身 | 先搞清 WHAT/WHY，再谈 HOW |
| 2. 逐层下降 | 一次一问，引导用户从当前层下降到 L3-L4 | 答不出开放题 → 给选项；没方向 → 资源盘点 |
| 3. 六问验证 | 确认 WHO / AS-IS / PAIN / WHY / DONE / LIMIT 都有答案 | 自然收集，缺失的再补问 |
| 4. 三重校验 | 可行性 + 矛盾检测 + 需求重构 | 发现问题温和指出，用户决定是否调整 |
| 5. 技术方案推介 | 推介 2-3 个方案，每个双层可读（大白话 + 技术细节），用户选定 | 对比表维度取用户关心的，不用技术术语 |
| 6. 文档产出 | 输出结构化需求文档，含选定方案和预期产物 | 格式见 references/output-template.md |
| 7. 移交 | **直接进入 `dev-go`** | 无需额外翻译 |

## 规则

- **HARD-GATE**：需求用户确认前，不提任何实现方案
- 一次只问一个问题，不连珠炮
- 检测到方案污染（"用多 agent 做 XX"）→ 先回到问题，方案留到 Phase 5
- 三重校验是诊断不是否决——发现问题提出来，用户决定
- 多个需求打包来 → 拆分，让用户排优先级，逐个走流程
